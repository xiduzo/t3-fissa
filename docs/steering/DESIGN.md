# Fissa Design Guidelines

> Generated by `steer-design`. Refine this document as the design system evolves — do not regenerate from scratch.

Last updated: 2026-04-17

## Design Principles

1. **Social first** — Design for the crowd, not the individual. Features should feel better when 10 people are in the room than when one is. Prioritise shared visibility over personal dashboards.
2. **Frictionless joining** — Guests must be inside a party within seconds. Every tap or screen added to the join flow is a cost. Optimise for the zero-to-in-the-queue path.
3. **Fun over serious** — Fissa celebrates the party moment. Use the full colour palette, emoji, and snarky copy to keep the tone light. Avoid the polished-product aesthetic of productivity apps — this is supposed to feel alive.
4. **Spotify-native** — Lean on Spotify's visual language where it appears (album art, track names). Don't redesign discovery or playback controls Spotify already owns.
5. **Host accountability** — Make host controls visible and decisive. When the host acts (skip, remove, pause), it should feel authoritative — not hidden in a hamburger menu.

## Tokens

### Color — Themes

Fissa uses dynamic party themes. The active theme is set per fissa and applies globally. Each theme defines three tiers via Tailwind class extensions.

| Theme          | 100 (light)   | 500 (primary) | 900 (dark)    | Gradient                  |
| -------------- | ------------- | ------------- | ------------- | ------------------------- |
| **Pinkey**     | `#FFCAF7`     | `#FF5FE5`     | `#150423`     | `#FF5FE5` → `#FF5F72`    |
| **Orangy**     | `#FFC9C9`     | `#FF5F5F`     | `#1C0A00`     | `#FF5F5F` → `#FF995F`    |
| **Greeny**     | `#CBFFE3`     | `#5FFF95`     | `#021600`     | `#5FFF95` → `#5FFFEC`    |
| **Blueey**     | `#CBF9FF`     | `#5FB2FF`     | `#001428`     | `#5FB2FF` → `#18FFF1`    |
| **Sunny**      | `#FFC9C9`     | `#FFAD33`     | `#241800`     | `#FFBF5F` → `#DFFF5F`    |
| **Limey**      | `#FFFED9`     | `#CCFF5F`     | `#0B1A00`     | `#FFF95F` → `#BCFF4E`    |
| **Extrodonary**| `#E8E8E8`     | `#8C8C8C`     | `#050505`     | `#8C8C8C` → `#B0B0B0`    |

**Usage rules:**
- `theme-100` — backgrounds, subtle fills
- `theme-500` — primary actions, highlights, active states
- `theme-900` — base dark background of a party screen
- Never hardcode a hex value that corresponds to a theme token. Use the Tailwind color extension (`bg-theme-500`, `text-theme-100`, etc.)

## UI States Required

Every interactive feature must handle all of these unless explicitly excluded in the design spec.

- **Default** — base state
- **Loading** — async operation in progress (skeleton or spinner)
- **Error** — something went wrong; show inline, not modal, unless data will be lost
- **Empty** — no data; use `EmptyState` component with relevant emoji and snarky copy
- **Disabled** — action not available in context; use `dimmed` or `disabled` prop on Button
- **Focus** — keyboard/remote navigation active (must have visible indicator)

## Copy & Tone

- **Snarky and warm** — write like the app is a good friend at a party, not a brand
- **Emoji are first-class** — use them in empty states, toasts, and section labels to lighten the tone
- **Short wins** — CTAs are 1–3 words max; toasts are one line
- **Avoid corporate language** — no "successfully completed", "please wait", or "an error occurred"

Example: `No tracks yet 🎶 Add something good.` not `No tracks have been added to the queue yet.`

## Accessibility Standards

Target: **WCAG 2.1 AA** minimum.

- Minimum contrast ratio: 4.5:1 for body text, 3:1 for large text and UI components
- All interactive elements keyboard/remote-navigable with visible focus indicator
- All images have descriptive `alt` text; decorative images use `alt=""`
- Color is never the sole means of conveying information (theme changes affect style, not meaning)
- Form inputs have associated labels, not just placeholder text
- Emoji used for decoration should be wrapped with `aria-hidden`; emoji conveying meaning need a text alternative

## Responsive Behavior

Fissa is primarily a mobile app (Expo). The web app is a marketing site + live fissa viewer.

| Surface           | Strategy                                      |
| ----------------- | --------------------------------------------- |
| Mobile (Expo)     | Mobile-first; single-column; full-screen pages |
| Web (marketing)   | Standard Tailwind breakpoints (sm/md/lg/xl); desktop-first hero |
| Web (fissa view)  | Responsive single-column; mirrors mobile layout |

No custom breakpoint overrides — Tailwind defaults apply.
