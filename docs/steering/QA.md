# Fissa QA Standards

> Generated by `steer-qa`. Refine this document as the test strategy evolves — do not regenerate from scratch.

Last updated: 2026-04-17

## Test Strategy

Tests exist in `packages/utils` today. The intent is to expand coverage to all service and repository layers. UI components and the Expo app are not currently tested — that is a known gap, not a standard.

| Level       | Type                    | Tool                         | Purpose                                           |
| ----------- | ----------------------- | ---------------------------- | ------------------------------------------------- |
| Unit        | Pure functions, utils   | Vitest                       | Sorting, formatting, array utilities              |
| Unit        | Services / repositories | Vitest + vitest-mock-extended | Business logic with mocked I/O boundaries        |
| Integration | API + DB                | Not yet set up               | End-to-end request → DB roundtrip                |
| E2E         | User flows              | Not yet set up               | Critical party join/vote/skip flows               |

## Coverage Requirements

No enforced thresholds exist today. The standard to apply when writing tests:

- **Business logic must be tested.** Anything in `packages/api/src/service/` and `packages/api/src/repository/` should have unit tests.
- **Pure functions must be tested.** Utilities in `packages/utils/` are the primary test surface — test all public exports.
- **UI components are not required.** No testing requirement on Expo or web components yet.
- **Bug fixes must include a regression test** that would have caught the bug before the fix.

> **Gap:** Coverage is not enforced in CI. This is a known gap — coverage thresholds and CI gating will be added as test coverage grows.

## Test Patterns

- **Framework:** Vitest 1.6.0 with `globals: true` (no need to import `describe`/`it`/`expect`)
- **Test file location:** Colocated with source — `*.test.ts` next to the file under test
- **Shared test utilities:** `@fissa/test` package — import from here, not directly from `vitest-mock-extended`
- **Partial test objects:** Use `fromPartial<T>()` from `@fissa/test` to construct typed test fixtures without filling every field
- **Mock library:** `vitest-mock-extended` (imported via `@fissa/test`) for interface mocks
- **Mock boundary:** Only mock at system boundaries — external APIs (Spotify), the database, the tRPC context. Never mock internal service or repository classes when testing other services.
- **Test order:** Tests run in shuffled order (`sequence: { shuffle: true }`) — tests must not depend on execution order
- **Mock cleanup:** `clearMocks: true` — mocks are reset between each test automatically

### Example pattern

```ts
import { fromPartial } from "@fissa/test";
import { mock } from "vitest-mock-extended";

const mockFissaRepo = mock<IFissaRepository>();
const service = new FissaService(mockFissaRepo, ...);

it("does the thing", () => {
  mockFissaRepo.findByPin.mockResolvedValue(fromPartial<Fissa>({ pin: "ABC" }));
  const result = await service.someMethod("ABC");
  expect(result).toBe(...);
});
```

## Definition of Done

Before any task is considered ready to merge:

- [ ] Build passes (`pnpm build`)
- [ ] Type-checking passes (`pnpm type-check`)
- [ ] Linting passes (`pnpm lint`)
- [ ] Tests pass (`pnpm test`) — _CI does not yet gate this; run it locally_
- [ ] Business logic changes include unit tests for the affected service/utility
- [ ] Bug fixes include a regression test
- [ ] No `describe.skip` / `it.skip` introduced without a linked issue

## Commands

```bash
# Run all tests
pnpm test

# Run tests for a specific package
cd packages/utils && pnpm vitest

# Run tests in watch mode
cd packages/utils && pnpm vitest --watch

# Run with coverage report
cd packages/utils && pnpm vitest --coverage
```

## Test Environments

| Environment | Purpose                  | Access                                      |
| ----------- | ------------------------ | ------------------------------------------- |
| Local       | Development testing      | `pnpm test` from repo root                  |
| CI          | Build + lint + type-check gate on PRs | GitHub Actions — `.github/workflows/ci.yml` |

> **Gap:** Tests are not currently gated in CI. The intent is that `pnpm test` runs as a required check on all PRs. This is a known gap to close.

## Known Flaky Areas

- **Spotify sync tests (if written):** The `FissaSyncOrchestrator` manages polling loops with time-dependent behaviour. Any tests touching this area should mock timers with `vi.useFakeTimers()` and avoid real `sleep()` calls.
- No existing skipped or known-flaky tests in the current suite.
