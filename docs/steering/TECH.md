## Architecture

**Monorepo layout:**

```
apps/
  expo/     React Native mobile app (Expo SDK 54)
  server/   Hono HTTP server ‚Äî hosts tRPC endpoint
  web/      Vite SPA ‚Äî marketing + live fissa view

packages/
  api/      tRPC router + services + repositories + DI container
  auth/     Better Auth configuration + session helpers
  db/       Drizzle schema, migrations, Postgres client (SERVER ONLY)
  env/      Zod-validated environment variables
  utils/    Shared utilities, custom error classes, Spotify helpers
  config/   Shared ESLint + Tailwind configs
```

**Patterns:**

- **Layered architecture:** tRPC procedures ‚Üí Services ‚Üí Repositories ‚Üí Drizzle/Postgres
- **Dependency injection:** `packages/api/src/container.ts` wires the full service graph per-request. No shared mutable state between requests.
- **Repository pattern:** All DB access through `IBadgeRepository`, `IFissaRepository`, `ITrackRepository`, `IVoteRepository`, `IUserRepository` interfaces.
- **Spotify sync loop:** `FissaSyncOrchestrator` runs background polling loops for active fissas. Each fissa has its own sync interval managed by the orchestrator.
- **State management (mobile):** Zustand stores in `packages/utils/stores/` for Spotify device/track/playlist state. Server state via TanStack Query + tRPC.

**Database schema tables:** `users`, `sessions`, `accounts`, `fissas`, `tracks`, `votes`, `badges`, `usersInFissas`, `userBadges`

## Key Constraints

- **`@fissa/db` is server-only.** The `packages/db` package instantiates a Postgres client at module load time. Never import from `@fissa/db` in Expo/React Native code. Import schema types directly from `packages/db/schema` if needed.
- **All external I/O behind interfaces.** `ISpotifyService`, `IFissaRepository`, etc. ‚Äî concrete implementations are injected via the DI container, never constructed directly in services.
- **DI container is per-request.** The container in `container.ts` is created fresh per tRPC context ‚Äî do not store references to services across requests.
- **Spotify is the only music source.** No abstraction for other providers ‚Äî Spotify-specific types and API calls are used throughout.
- **Environment variables validated at startup.** All env vars go through `packages/env` (Zod). Adding a new var requires updating the schema there.

## Code Conventions

- **Naming:** PascalCase components and classes, camelCase functions/variables, kebab-case files (except React components)
- **File structure:** Feature-based folders within apps; packages are domain-scoped (`api`, `db`, `auth`, `utils`)
- **Barrel exports:** Each package/folder has an `index.ts` re-exporting its public surface
- **Test files:** Colocated alongside source (e.g. `array.test.ts` next to `array.ts`) in `packages/utils`
- **tRPC procedures:** Defined in `packages/api/src/router/`, one file per domain (`fissa.ts`, `track.ts`, `vote.ts`, `auth.ts`)
- **Zod schemas for procedure inputs:** Shared constants in `packages/api/src/router/constants.ts` (e.g. `Z_PIN`, `Z_TRACK_ID`)
- **Custom errors:** Domain errors extend classes in `packages/utils/classes/Error.ts` (e.g. `NotTheHost`, `FissaIsPaused`)

## Architecture Decision Records

| Decision              | Rationale                                                                 | Date       |
| --------------------- | ------------------------------------------------------------------------- | ---------- |
| tRPC for API layer    | End-to-end TypeScript type safety from server to web and mobile clients   | ‚Äî          |
| Drizzle ORM           | Type-safe SQL queries without codegen overhead; schema-as-code            | ‚Äî          |
| Hono as HTTP server   | Lightweight, fast, first-class TypeScript ‚Äî good fit for tRPC adapter     | ‚Äî          |
| Expo for mobile       | Cross-platform iOS/Android from one codebase; managed workflow            | ‚Äî          |
| Repository pattern    | Decouples business logic from DB; enables testing without a real database | ‚Äî          |
| DI container per req  | Avoids shared mutable state; makes unit testing straightforward           | ‚Äî          |
| Turborepo             | Parallel task execution and caching across the monorepo                   | ‚Äî          |

## Known Pain Points

- **Spotify sync is fragile.** `FissaSyncOrchestrator` manages per-fissa background loops. Spotify's playback API has eventual consistency ‚Äî track position and playback state can lag. Changes here require careful testing against real Spotify devices, not mocks.
- **Expo + pnpm:** `.npmrc` contains a workaround for pnpm/Expo compatibility. Be cautious upgrading Expo or pnpm without testing the full mobile build.
